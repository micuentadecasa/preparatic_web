🎯 Introducción a las Bases de Datos Relacionales

El modelo relacional, propuesto por Edgar F. Codd en 1970, es la base de la mayoría de los sistemas de gestión de bases de datos (SGBD) modernos. Este modelo organiza los datos en tablas relacionadas entre sí, proporcionando integridad y consistencia de los datos.

⚡ Conceptos Fundamentales para el Examen

  • Reglas de Codd: 12 reglas que definen un sistema relacional puro
  • Normalización: Proceso para eliminar redundancias y anomalías
  • Formas Normales: Criterios para evaluar el diseño de tablas
  • SQL: Lenguaje estándar para gestión de bases de datos relacionales
  • ACID: Propiedades de transacciones (Atomicity, Consistency, Isolation, Durability)

💡 Tips para el Examen

📘 Las 12 Reglas de Codd

Edgar F. Codd definió 12 reglas que un sistema debe cumplir para ser considerado relacional. Estas reglas son fundamentales para entender las bases de datos relacionales.

Regla 0: Fundación

Un sistema RDBMS debe gestionar sus bases de datos enteramente mediante sus capacidades relacionales.

Regla 1: Información

Toda la información en una base de datos relacional se representa explícitamente en los valores almacenados en las tablas.

Regla 2: Acceso Garantizado

Cada dato se puede acceder lógicamente utilizando una combinación de nombre de tabla, nombre de columna y valor de clave primaria.

📋 Reglas 3-6 de Codd

Regla 3: Tratamiento Sistemático de Valores Nulos

El SGBD debe permitir valores nulos para representar información desconocida o no aplicable, y debe manejarlos sistemáticamente.

Regla 4: Diccionario de Datos Activo en Línea

La descripción de la base de datos se almacena en línea en tablas relacionales, accesibles a usuarios autorizados mediante el mismo lenguaje relacional.

Regla 5: Sublenguaje de Datos Completo

Debe existir al menos un lenguaje relacional que pueda utilizarse para definir, manipular y restringir datos, tanto de forma interactiva como en programas.

Regla 6: Actualización de Vistas

Todas las vistas que son teóricamente actualizables deben ser actualizables por el sistema.

  • Entiende la importancia de cada regla en la integridad de datos
  • Reconoce sistemas que cumplen o no cumplen las reglas

📘 Normalización de Bases de Datos

La normalización es el proceso de organizar los datos en una base de datos para minimizar la redundancia y mejorar la integridad. Es un tema crítico para exámenes de bases de datos.

📊 Objetivos de la Normalización

La normalización busca eliminar tres tipos de anomalías:

Anomalía Descripción Ejemplo
Inserción No se puede insertar cierta información sin otra No se puede insertar un cliente sin pedidos
Actualización Cambios deben realizarse en múltiples lugares Actualizar dirección de cliente en múltiples pedidos
Eliminación Eliminar datos importantes al borrar otros Perder datos de cliente al eliminar su último pedido

💡 Error Común en Exámenes

Confundir las anomalías de inserción, actualización y eliminación.

Explicación: Anomalía de inserción: no poder insertar datos sin otros. Anomalía de actualización: tener que cambiar datos en múltiples lugares. Anomalía de eliminación: perder información importante al borrar otros datos.

📋 Pasos de la Normalización

⚠️ Consideraciones Importantes

El proceso de normalización sigue estos pasos sistemáticos:

Paso 1: Eliminar grupos repetitivos (1FN)
Paso 2: Crear tablas separadas para conjuntos de valores que apliquen a múltiples registros (2FN)
Paso 3: Eliminar dependencias transitivas (3FN)
Paso 4: Aislar dependencias multivaluadas (4FN)
Paso 5: Eliminar dependencias de unión (5FN)

✅ Ejemplo Práctico

Tabla ClientesPedidos con campos: ClienteID, Nombre, Teléfono, PedidoID, Producto, Cantidad

  • 1FN: Separar grupos repetitivos
  • 2FN: Eliminar parciales dependencias de clave primaria
  • 3FN: Eliminar dependencias transitivas

📘 Formas Normales Detalladas

Las formas normales son criterios sucesivos para evaluar la estructura de las tablas en una base de datos relacional. Cada forma normal corrige ciertos tipos de redundancia.

📋 Primera Forma Normal (1FN)

⚠️ Requisitos de la 1FN

Una tabla está en 1FN si:

  • Todos los atributos contienen valores atómicos (indivisibles)
  • No hay grupos repetitivos
  • Cada fila es única

✅ Ejemplo de 1FN

Tabla incorrecta: Estudiantes con campo "Teléfonos" que contiene "912345678, 987654321"

Tabla correcta en 1FN: Cada teléfono en una fila separada con clave compuesta

📊 Segunda Forma Normal (2FN)

⚠️ Requisitos de la 2FN

Una tabla está en 2FN si:

  • Está en 1FN
  • No tiene dependencias parciales (atributos no clave dependen de parte de la clave primaria)

💡 Escenario Práctico

Tabla Pedidos con campos: PedidoID, ProductoID, ClienteID, NombreCliente, FechaPedido

Solución: Separar en tablas Pedidos (PedidoID, ClienteID, FechaPedido) y Clientes (ClienteID, NombreCliente)

📋 Tercera Forma Normal (3FN)

⚠️ Requisitos de la 3FN

Una tabla está en 3FN si:

  • Está en 2FN
  • No tiene dependencias transitivas (A→B y B→C, entonces A→C)

✅ Ejemplo de 3FN

Tabla incorrecta: Empleados (EmpleadoID, Nombre, DepartamentoID, NombreDepartamento)

Tabla correcta: Empleados (EmpleadoID, Nombre, DepartamentoID) y Departamentos (DepartamentoID, NombreDepartamento)

❓ Preguntas Frecuentes de Examen

¿Cuál es la diferencia entre 2FN y 3FN?

2FN elimina dependencias parciales (parte de la clave determina un atributo), mientras que 3FN elimina dependencias transitivas (A→B y B→C implica A→C).

¿Qué es una dependencia transitiva?

Ocurre cuando un atributo no clave depende de otro atributo no clave. Por ejemplo, si ClienteID→Ciudad y Ciudad→Región, entonces ClienteID→Región transitivamente.
