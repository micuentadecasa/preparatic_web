🏃‍♂️ Scrum: Framework ágil para gestionar desarrollo de productos complejos, basado en equipos auto-organizados que entregan valor en iteraciones cortas llamadas Sprints.

👥 Los 3 Roles de Scrum

🎯 Product Owner (PO)

Responsabilidades:

  • Definir y priorizar Product Backlog
  • Escribir User Stories claras
  • Aceptar o rechazar incrementos
  • Comunicar visión del producto
  • Interactuar con stakeholders
  • Maximizar valor del producto
⚠️ Errores Comunes:
  • Product Owner por comité
  • Falta de disponibilidad
  • Micromanagement del equipo
  • Cambios constantes de prioridades

🏃‍♂️ Scrum Master (SM)

Responsabilidades:

  • Facilitar eventos de Scrum
  • Eliminar impedimentos del equipo
  • Entrenar al equipo en Scrum
  • Proteger al equipo de interrupciones
  • Promover mejora continua
  • Servant leadership
🎯 Skills Clave:
  • Facilitación de grupos
  • Coaching y mentoring
  • Resolución de conflictos
  • Pensamiento sistémico

👨‍💻 Development Team

Características:

  • Tamaño: 3-9 miembros idealmente
  • Cross-functional: Todas skills necesarias
  • Self-organizing: Deciden cómo hacer el trabajo
  • No jerarquías: Todos son desarrolladores
  • Accountability: Responsabilidad colectiva

📦 Los 3 Artefactos de Scrum

📝 Product Backlog

Definición: Lista ordenada y emergente de todo lo que podría ser necesario en el producto.

  • Propietario: Product Owner
  • Características: Priorizado, refinado, estimado
  • Formato: User Stories, Epics, Tasks
  • Commitment: Product Goal
📋 Criterios INVEST para User Stories:
  • Independent: No depende de otras
  • Negotiable: Flexible en implementación
  • Valuable: Aporta valor al usuario
  • Estimable: Se puede estimar esfuerzo
  • Small: Tamaño manejable
  • Testable: Criterios aceptación claros

🎯 Sprint Backlog

Definición: Objetivo del Sprint más elementos del Product Backlog seleccionados más plan para entregarlos.

  • Propietario: Development Team
  • Características: Plan detallado, emergente, visible
  • Actualizaciones: Diarias durante Daily Scrum
  • Commitment: Sprint Goal

🚀 Increment

Definición: Suma de todos los elementos completados durante el Sprint y valor de todos los incrementos anteriores.

  • Características: "Done", usable, potencialmente releaseable
  • Definition of Done: Criterios de completitud
  • Compromiso: Definition of Done

📋 Kanban: Método para gestionar y mejorar el flujo de trabajo, visualizando el proceso, limitando el trabajo en progreso y optimizando continuamente.

🎯 Los 6 Principios de Kanban

🔄 Principios de Gestión del Cambio

  • 1. Empezar con lo que haces ahora: No cambios dramáticos iniciales
  • 2. Acordar buscar mejora continua: Evolución constante
  • 3. Respetar procesos y roles actuales: Cambio evolutivo

🎯 Principios de Entrega de Servicios

  • 4. Entender y enfocarse en necesidades del cliente: Valor customer-first
  • 5. Gestionar el trabajo, no a las personas: Autoorganización
  • 6. Evolucionar políticas para mejorar resultados: Adaptación basada en datos

🏗️ Las 6 Prácticas Centrales de Kanban

👁️ 1. Visualizar el Flujo de Trabajo

Hacer visible todo el trabajo y proceso mediante un tablero Kanban.

  • Columnas representan estados del proceso
  • Tarjetas representan elementos de trabajo
  • Swimlanes para diferentes tipos de trabajo
  • Colores para clasificación/prioridad

⚖️ 2. Limitar el Trabajo en Progreso (WIP)

Establecer límites para la cantidad de trabajo simultáneo.

  • Beneficios: Reduce multitasking, mejora flujo
  • Implementación: Límites por columna/persona
  • Regla: Pull system - no push
  • Objetivo: Optimizar throughput, no utilización

📏 3. Gestionar el Flujo

Monitorizar, medir y reportar el flujo de trabajo.

  • Lead Time: Tiempo total en el sistema
  • Cycle Time: Tiempo de trabajo activo
  • Throughput: Items completados por período
  • WIP Age: Tiempo items en progreso

📝 4. Hacer Políticas Explícitas

Definir claramente cuándo y cómo se mueve el trabajo.

  • Definition of Done: Criterios completitud
  • Definition of Ready: Criterios para empezar
  • Políticas de Priorización: Cómo ordenar
  • Políticas de Bloqueos: Cómo manejar impedimentos

🔄 5. Implementar Loops de Feedback

Crear mecanismos regulares de inspección y adaptación.

  • Standups diarios: Coordinación del equipo
  • Service Delivery Review: Métricas y flujo
  • Operations Review: Capacidad y demanda
  • Risk Review: Identificación de riesgos

🚀 6. Mejorar Colaborativamente

Evolucionar experimentalmente usando modelos/métodos científicos.

  • Hipótesis-driven: Cambios basados en teorías
  • Experimentos: A/B testing de procesos
  • Medición: Datos para validar hipótesis
  • Evolutionary change: Pequeños cambios frecuentes

📐 Técnicas de Estimación Ágil

🎲 Planning Poker

Proceso:

  1. Product Owner presenta User Story
  2. Equipo hace preguntas de clarification
  3. Cada miembro selecciona carta secretamente
  4. Revelan cartas simultáneamente
  5. Si hay consenso → estimación final
  6. Si discrepancia → discusión y nueva ronda
🃏 Secuencia Fibonacci Modificada: 0, ½, 1, 2, 3, 5, 8, 13, 20, 40, 100, ∞, ?

👔 T-Shirt Sizing

Escala: XS, S, M, L, XL, XXL

Beneficios:

  • Rápido para estimaciones iniciales
  • Menos precisión → menos ansiedad
  • Útil para epics y features grandes
  • Fácil de entender por stakeholders

📊 Affinity Estimating

Proceso:

  1. Escribir todas las stories en sticky notes
  2. Ordenar por complejidad relativa
  3. Agrupar stories de similar tamaño
  4. Asignar puntos a cada grupo

Ideal para: Muchas stories (50+) en poco tiempo

⚡ No Estimates (#NoEstimates)

Filosofía: Enfocarse en flujo y throughput en vez de estimaciones

Métricas:

  • Story count por sprint
  • Cycle time por story
  • Throughput histórico
  • Forecasting probabilístico

🎯 User Story Mapping

Técnica de Jeff Patton para organizar y priorizar funcionalidades desde perspectiva del usuario.

📋 Proceso de Story Mapping

  1. Frame: Define usuarios y objetivos
  2. Map: Documenta user journey
  3. Explore: Detalla actividades y tasks
  4. Slice: Identifica releases y MVPs
  5. Plan: Prioriza y organiza development

🏗️ Estructura del Map

  • Backbone: Activities (nivel alto)
  • Walking skeleton: Tasks (nivel detalle)
  • Horizontal slices: Releases/MVPs
  • Vertical slices: User journeys
✅ Beneficios del Story Mapping:
  • Visión holística del producto
  • Identificación de gaps en funcionalidad
  • Priorización basada en user value
  • Comunicación efectiva con stakeholders
  • Planning de releases coherente

📊 Forecasting y Predictibilidad

📈 Velocity-based Forecasting

Formula básica:

Sprints Restantes = Story Points Restantes / Velocity Promedio
  • Usar últimos 3-6 sprints para promedio
  • Considerar variabilidad (min/max)
  • Ajustar por holidays y vacaciones
  • Re-forecast cada sprint

🎲 Monte Carlo Simulation

Enfoque probabilístico para forecasting más preciso.

  • Input: Velocity histórica y backlog size
  • Simulation: Miles de scenarios posibles
  • Output: Probabilidades de completion dates
  • Beneficio: Ranges en vez de dates fijas

🏢 SAFe (Scaled Agile Framework)

Framework más popular para escalado ágil, usado por grandes corporations worldwide.

🎯 Essential SAFe

Nivel básico: 1 Agile Release Train (ART)

  • Equipos: 3-9 Agile Teams
  • Roles: Release Train Engineer, Product Manager
  • Eventos: PI Planning, System Demo, I&A
  • Artifacts: Program Backlog, Program Increment

🏗️ Large Solution SAFe

Para sistemas complejos: Múltiples ARTs + Solution Train

  • Roles adicionales: Solution Architect, Solution Manager
  • Eventos: Solution Demo, Pre-PI Planning
  • Artifacts: Solution Backlog, Solution Increment

🏛️ Portfolio SAFe

Nivel estratégico: Alineación con business strategy

  • Lean Portfolio Management: Strategy, Investment, Agile PMO
  • Value Streams: End-to-end flow de value
  • Budgeting: Lean budgets vs project-based

🎯 Full SAFe

Framework completo: Todos los niveles integrados

  • Spanning Palette con prácticas adicionales
  • Foundational principles y core values
  • Implementation roadmap

🚂 Agile Release Train (ART)

Core concept de SAFe: Long-lived team of Agile teams

👥 Roles del ART

  • Release Train Engineer: Chief Scrum Master
  • Product Manager: What and when
  • System Architect: Technical guidance
  • Business Owners: Key stakeholders
  • Scrum Masters: Team level facilitation
  • Product Owners: Team backlogs

🔄 PI Planning Event

Duración: 2 días cada 8-12 semanas

  1. Business Context: Vision y priorities
  2. Product/Solution Vision: Roadmap
  3. Architecture Vision: Technical direction
  4. Planning Breakouts: Team planning
  5. Draft Board Review: Dependencies
  6. Management Review: Final adjustments
  7. Planning Retrospective: Process improvement

🔗 Nexus

By Ken Schwaber: Extension de Scrum para 3-9 teams

🏗️ Nexus Framework

  • Nexus Integration Team: PO, SM, devs
  • Nexus Sprint Planning: Dependencies y integration
  • Nexus Daily Scrum: Integration issues
  • Nexus Sprint Review: Integrated increment
  • Nexus Retrospective: Cross-team improvements
✅ Nexus Focus: Integration y dependencies entre teams manteniendo simplicidad de Scrum.

🎵 Spotify Model

Autonomous teams organizados en tribes, squads, guilds y chapters

🏃‍♂️ Squad (Team)

  • 6-12 personas
  • Cross-functional
  • Own mission
  • Minimal dependencies

🏘️ Tribe (Collection of squads)

  • <100 personas
  • Shared area de negocio
  • Minimal dependencies
  • Tribe lead facilita

👥 Chapter & Guild

  • Chapter: Same skills, same tribe
  • Guild: Same interest, any tribe
  • Knowledge sharing
  • Communities of practice
⚠️ Spotify Model Reality: Spotify evolucionó más allá del model original. No es framework prescriptivo sino inspiration.

🚧 Desafíos Comunes del Escalado

🔗 Dependencies

Problema: Múltiples teams working en shared codebase

Solutions:

  • Architecture que minimiza dependencies
  • Dependency mapping y tracking
  • Cross-team planning sessions
  • Shared services y APIs

📢 Communication

Problema: Information silos y coordination overhead

Solutions:

  • Communities of practice
  • Regular all-hands meetings
  • Shared dashboards y metrics
  • Documentation standards

⚖️ Governance

Problema: Balance entre autonomy y alignment

Solutions:

  • Clear decision rights
  • Architectural guardrails
  • Portfolio-level prioritization
  • Outcome-based governance

🏗️ Technical Debt

Problema: Accumulated debt across multiple teams

Solutions:

  • Dedicated refactoring sprints
  • Technical debt backlog
  • Code quality metrics
  • Architectural review boards

🚧 Common Pitfalls y Cómo Evitarlos

❌ "Agile Theater"

Problema: Doing Agile sin being Agile

Síntomas:

  • Ceremonies sin purpose
  • No empowerment real
  • Command & control mindset
  • Focus en process sobre outcomes

Prevention:

  • Culture change before process change
  • Leadership modeling behaviors
  • Outcome-based metrics
  • Regular culture assessments

⚡ "Big Bang" Implementation

Problema: Cambiar toda organization de una vez

Riesgos:

  • Change overload
  • Resistance escalation
  • Quality degradation
  • Business disruption

Alternative:

  • Pilot approach con 1-2 teams
  • Learn y adapt antes de scaling
  • Wave-based rollout
  • Parallel track con traditional

🛠️ "Tool-First" Mentality

Problema: Focusing en tools antes que people

Issues:

  • Over-engineering workflows
  • Tool becomes constraint
  • Missing human elements
  • Configuration complexity

Better approach:

  • Start con simple tools
  • Process clarity first
  • Tool selection based en needs
  • Regular tool assessment

📊 Metrics Misuse

Problema: Using metrics para punishment vs improvement

Anti-patterns:

  • Individual velocity tracking
  • Velocity comparison between teams
  • Burndown micromanagement
  • Estimate accuracy pressure

Healthy metrics:

  • Team-based measurements
  • Trend analysis over absolute values
  • Outcome metrics prioritized
  • Metrics for teams' self-improvement
