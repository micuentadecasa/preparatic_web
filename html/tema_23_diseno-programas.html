🎯 Introducción al Tema

El diseño de programas es una fase crítica en el desarrollo de software que se centra en cómo se estructurará y organizará el código para implementar los requisitos especificados. Este tema abarca conceptos fundamentales como el diseño estructurado, los enfoques de análisis de transformación y transacción, y los principios de cohesión y acoplamiento que guían una arquitectura de software robusta y mantenible.

📘 Conceptos Clave

  • Diseño estructurado basado en estructuras de control
  • Análisis de transformación para sistemas de procesamiento
  • Análisis de transacción para sistemas interactivos

⚡ Principios Fundamentales

  • Alta cohesión dentro de los módulos
  • Bajo acoplamiento entre módulos
  • Monentrada-monosalia en módulos

🎯 Objetivos del Diseño

  • Facilitar mantenimiento y evolución
  • Reducir complejidad del sistema
  • Mejorar calidad y confiabilidad

💡 Tips para el Examen

Memoriza los tipos de cohesión en orden de calidad (funcional > secuencial > comunicacional > procedural > temporal > lógico > coincidental). Entiende cuándo aplicar el análisis de transformación vs. transacción. El examen suele incluir preguntas sobre identificar tipos de cohesión en fragmentos de código.

🔥 HOT TIP: El diseño estructurado se basa en el teorema de la estructura que establece que cualquier programa puede construirse usando solo tres estructuras de control básicas: secuencia, selección e iteración.

📘 Fundamentos del Diseño Estructurado

El diseño estructurado es una técnica de diseño que se basa en el teorema de la estructura que establece que cualquier programa puede construirse usando solo tres estructuras de control básicas: secuencia, selección e iteración. Esta metodología promueve la creación de programas modulares, legibles y mantenibles.

🧩 Principios del Diseño Estructurado

El diseño estructurado se fundamenta en la descomposición jerárquica del problema, donde se divide en subproblemas más pequeños y manejables. Cada subproblema se resuelve mediante módulos que se comunican a través de interfaces bien definidas.

⚙️ Estructuras de Control Básicas

Secuencia (ejecución lineal), Selección (bifurcación condicional) e Iteración (bucles) son las únicas estructuras necesarias para expresar cualquier algoritmo. Esta simplicidad facilita la comprensión, prueba y mantenimiento del código.

🏆 Beneficios del Enfoque Estructurado

Incluye mayor legibilidad del código, facilidad para la depuración, mejor mantenibilidad, y reducción de errores lógicos complejos. Además, facilita el trabajo en equipo al establecer estándares claros de codificación.

📋 Técnicas del Diseño Estructurado

Las técnicas del diseño estructurado incluyen el uso de diagramas de estructura (structure charts) para representar la organización modular del sistema, y la definición de interfaces claras entre módulos para controlar el flujo de datos y control.

💡 Puntos Importantes para el Examen

  • Los módulos deben tener una única entrada y una única salida (monentrada-monosalia)
  • Evitar el uso de GOTO y estructuras de control no estructuradas
  • Aplicar el principio de "divide y vencerás" para la descomposición funcional

P: ¿Cuál es la característica principal de un módulo en diseño estructurado?

🔥 HOT TIP: Análisis de transformación para sistemas que transforman entradas en salidas (procesamiento por lotes). Análisis de transacción para sistemas que seleccionan entre múltiples caminos de procesamiento (sistemas interactivos).

📘 Análisis de Transformación

El análisis de transformación es una técnica utilizada cuando el sistema puede modelarse como una transformación de entradas en salidas. Es aplicable cuando existe un flujo de datos que pasa por una serie de transformaciones hasta producir resultados.

📊 Análisis de Transacción

El análisis de transacción se aplica cuando el sistema recibe una entrada que determina qué camino de procesamiento seguir. Es común en sistemas que manejan múltiples tipos de transacciones, donde cada transacción requiere un tratamiento específico diferente.

🔄 Comparación de Técnicas

Característica Análisis de Transformación Análisis de Transacción
Modelo de Flujo Entrada → Procesamiento → Salida Entrada → Selección de Camino → Procesamiento
Estructura Lineal y secuencial Jerárquica con ramificaciones
Aplicación Típica Procesamiento por lotes, cálculos Sistemas interactivos, menús

💡 Error Común en Exámenes

Confundir cuándo aplicar cada tipo de análisis. El análisis de transformación es para sistemas que transforman datos, mientras que el de transacción es para sistemas que seleccionan entre caminos alternativos.

Explicación: El análisis de transformación se usa cuando hay un flujo lineal de datos a través de transformaciones sucesivas. El análisis de transacción se usa cuando se necesita tomar decisiones basadas en la entrada para seleccionar un camino de procesamiento.

🔥 HOT TIP: Cohesión funcional (mejor) - todos los elementos contribuyen a una única función. Acoplamiento de datos (mejor) - módulos pasan datos simples como parámetros.

📘 Cohesión: Medida de Fuerza de los Módulos

La cohesión es una medida de la fuerza de relación entre los elementos de un módulo. Una alta cohesión indica que los elementos del módulo están estrechamente relacionados entre sí y contribuyen a un propósito único y bien definido. Es un atributo deseable en el diseño de software.

🎯 Tipos de Cohesión (de mejor a peor)

  • Funcional: Todos los elementos contribuyen a una única función
  • Secuencial: Salida de uno es entrada del siguiente
  • Comunicacional: Elementos operan sobre los mismos datos
  • Procedural: Elementos relacionados por procedimiento de ejecución
  • Temporal: Elementos se ejecutan en el mismo periodo de tiempo
  • Lógica: Elementos realizan funciones lógicamente similares
  • Coincidental: Elementos no tienen relación aparente

📋 Ejemplos Prácticos

  • Funcional: Módulo que calcula impuestos
  • Secuencial: Módulo que lee, valida y procesa datos
  • Comunicacional: Módulo que actualiza varios campos de un registro
  • Coincidental: Módulo con funciones de impresión y cálculo sin relación

⚠️ Consideraciones Importantes

La cohesión funcional es el objetivo ideal, mientras que la cohesión coincidental es la menos deseable. El diseño debe buscar maximizar la cohesión dentro de cada módulo para mejorar la mantenibilidad y comprensión del sistema.

📘 Acoplamiento: Medida de Interdependencia

El acoplamiento es una medida de la interdependencia entre módulos. Un bajo acoplamiento es deseable ya que facilita el mantenimiento, prueba e independencia de módulos. Se refiere a cómo se comunican los módulos entre sí y cuánto conocen de otros módulos.

🎯 Tipos de Acoplamiento (de peor a mejor)

  • Contenido: Un módulo modifica el código interno de otro
  • Común: Módulos comparten variables globales
  • Externo: Módulos comparten base de datos global
  • Control: Un módulo controla el flujo de otro
  • Marca: Módulos comparten partes de una estructura
  • Datos: Módulos pasan datos simples como parámetros
  • Datos con Estructuras: Módulos pasan estructuras como parámetros

📋 Ejemplos Prácticos

  • Datos: Pasar un entero como parámetro
  • Marca: Pasar un objeto y acceder a sus campos
  • Control: Pasar un flag que determina el flujo
  • Común: Usar variables globales para compartir datos

⚠️ Consideraciones Importantes

El objetivo del diseño es minimizar el acoplamiento entre módulos. Un bajo acoplamiento significa que los cambios en un módulo tienen menor probabilidad de afectar a otros módulos, mejorando la mantenibilidad del sistema.

🔥 HOT TIP: Relaciona estos conceptos con principios SOLID de programación orientada a objetos: cohesión con Single Responsibility Principle y acoplamiento con Dependency Inversion Principle.

❓ Preguntas Frecuentes de Examen

¿Cuál es la diferencia entre cohesión y acoplamiento?

La cohesión mide qué tan relacionados son los elementos DENTRO de un módulo, mientras que el acoplamiento mide qué tan dependientes son los módulos ENTRE SÍ. Alta cohesión y bajo acoplamiento son objetivos del buen diseño.

¿Por qué es importante la cohesión funcional?

Porque cada módulo realiza una única función bien definida, lo que facilita la comprensión, prueba, mantenimiento y reutilización del código. Es el tipo más fuerte y deseable de cohesión.
