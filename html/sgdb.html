¿Qué es una Base de Datos?

Base de Datos (BD): Conjunto de datos almacenado en un soporte informático no volátil. Los datos deben estar estructurados e interrelacionados conforme a un modelo capaz de recoger el máximo contenido semántico.

Sistema de Gestión de Bases de Datos (SGBD)

SGBD: Conjunto de programas, procedimientos y lenguajes que permiten almacenar, actualizar y consultar datos contenidos en una BD, manteniendo su integridad, confidencialidad y seguridad.

Características Principales de un SGBD

  • Modelo de datos soportado
  • Uso de lenguajes de alto nivel (DDL, DML, DCL)
  • Acceso concurrente con gestión de bloqueos
  • Independencia física y lógica
  • Redundancia controlada de datos
  • Integridad y consistencia
  • Seguridad y control de acceso
  • Alto rendimiento funcional
  • Capacidad para alto volumen de datos

Modelo de Referencia ANSI/X3/SPARC

Modelo de 3 Niveles (1975)

El Database Architecture Framework Task Group (DAFTG) de ANSI/X3/SPARC definió una arquitectura estándar con 3 niveles diferenciados de representación de la información.

🔧 Nivel Interno o Físico

Se encarga de los aspectos más internos relacionados con el servidor (sistema operativo y gestión de ficheros). Objetivos:

  • Mejor tiempo de respuesta
  • Minimizar espacio de almacenamiento
  • Evitar redundancia de información
📋 Nivel Conceptual

Materializa la representación de los datos independientemente de su estructura física. Define el Esquema de BD:

  • Entidades y atributos
  • Interrelaciones entre entidades
  • Restricciones de integridad
  • Modelo de datos específico
👁️ Nivel Externo o Lógico

"Filtra" los esquemas de BD conforme a la parte que es de interés para un usuario concreto. Un esquema externo corresponde a una Vista.

Familias de Modelos de Datos

Transacciones y Propiedades ACID

¿Qué es una Transacción?

Transacción: Unidad elemental de trabajo delimitada por las sentencias Begin-transaction (comienzo) y Commit/Rollback (final exitoso o fallido). Entre ambas se sitúan las sentencias Read, Write a ejecutar como un bloque único.

🔗 Atomicidad (Atomicity)

La transacción debe tener efecto en su totalidad, no se permite ejecución parcial. Si se realiza COMMIT, todas las instrucciones se completan; si se realiza ROLLBACK, ninguna se completa.

✅ Consistencia (Consistency)

Sólo se ejecutan operaciones que respetan las reglas de integridad de la BD. Se asegura que los datos sean exactos y los esperados cuando un usuario los consulte.

🔒 Aislamiento (Isolation)

Una operación no afectará a las demás. Dos transacciones sobre la misma información son independientes. Los cambios no son visibles hasta ejecutar Commit.

💾 Durabilidad (Durability)

Una vez ejecutado el Commit, los cambios se hacen persistentes y sobreviven a fallos del sistema.

⚖️ Teorema CAP (Conjetura de Brewer)

En un SGBD distribuido no se pueden garantizar simultáneamente más que 2 aspectos de entre los 3 siguientes:

  • Consistencia (C): Cualquier lectura recibe la escritura más reciente o un error
  • Disponibilidad (A): Cualquier petición recibe respuesta no errónea
  • Tolerancia al Particionado (P): El sistema funciona aunque se descarten mensajes entre nodos

Control de Concurrencia

Concepto de Concurrencia

Concurrencia: Intercalación de operaciones de distintas transacciones para mejorar el rendimiento, asegurando siempre la integridad y consistencia de la BD.

⚠️ Problemas de Interferencias entre Transacciones

  • Lectura Sucia: T1 lee dato modificado por T2 antes del Commit de T2
  • Lectura No Repetible: T1 lee dos veces un valor y no coinciden porque T2 lo modificó
  • Lectura Fantasma: T2 inserta datos entre consultas iguales de T1

🔐 Two-Phase-Locking (2PL)

Mecanismo de control de concurrencia usado por la mayoría de SGBD:

  • Fase 1: Se adquieren los recursos necesarios
  • Fase 2: Se liberan los recursos
  • Los recursos solo se liberan después de Commit o Rollback

Aspectos que debe Garantizar un SGBD

  • Concurrencia: Accesos simultáneos sin conflictos
  • Consistencia: Valores sin contradicciones
  • Integridad: Valores conformes a reglas semánticas
  • Recuperación: Vuelta a estado íntegro tras fallo
  • Privacidad: Acceso según privilegios definidos

Componentes Principales del SGBD

  • Procesador de Consultas (QP): Analiza sintaxis y semántica, optimiza consultas
  • Gestor de Transacciones (TM): Ordena operaciones de transacciones concurrentes
  • Planificador (SC): Controla concurrencia y planifica ejecución
  • Gestor de Recuperación (RM): Ejecuta Commit y Rollback
  • Gestor de Buffers: Ejecuta operaciones Read y Write

📋 Evolución DAFTG (1985)

ANSI evolucionó al modelo DAFTG (RM) con 2 partes diferenciadas:

  • Definición de datos: DBA define diccionario y procesos
  • Manipulación de datos: Datos, transformaciones y usuarios finales

🔧 Componentes DAFTG RM

  • Data Management Tools (DMT): Herramientas de tratamiento
  • Data Mapping Control System (DMCS): Core del SGBD
  • Data Language Interface (DL): Interfaz general
  • Internal Data Language Interface (i-DL): Interfaz con SO

👤 Ampliación UFRM (1988)

Se añadieron componentes del User Facility Reference Model:

  • User Facility (UF): Conexión usuario-herramientas
  • User Data Language (UDL): Interfaz usuario-UF
  • Internal User Data Language (i-UDL): Interfaz UF-DMT

💡 Ventajas del Modelo ANSI

  • Independencia física y lógica de datos
  • Flexibilidad ante cambios en estructura
  • Separación de responsabilidades
  • Estandardización de interfaces
  • Facilita mantenimiento y evolución

🎯 Consideraciones de Diseño

Al diseñar un sistema de BD, considerar:

  • Modelo de datos más apropiado
  • Requisitos de concurrencia
  • Necesidades de recuperación
  • Políticas de seguridad
  • Escalabilidad futura
  • Integración con aplicaciones existentes
