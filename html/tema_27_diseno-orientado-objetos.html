🎯 Importancia del Diseño OO

El diseño orientado a objetos permite crear sistemas mantenibles, extensibles y reutilizables mediante la aplicación de principios fundamentales como la separación de responsabilidades, el bajo acoplamiento y la alta cohesión. UML proporciona el lenguaje estándar para modelar estos sistemas, mientras que los patrones ofrecen soluciones probadas a problemas recurrentes.

UML (Unified Modeling Language)

🏗️ Lenguaje de Modelado Unificado

UML es un lenguaje estándar para especificar, visualizar, construir y documentar sistemas de software. Se divide en 14 tipos de diagramas organizados en 3 categorías principales.

14 Tipos de Diagramas
7 Diagramas Estructurales
7 Diagramas Comportamiento
3 Diagramas Interacción

🏛️ Diagramas Estructurales (7 tipos)

Muestran la estructura estática del sistema: elementos que deben estar presentes en el sistema modelado.

1. Diagrama de Clases

Propósito: Mostrar clases del sistema y relaciones entre ellas

  • Elementos: Clases, interfaces, atributos, métodos
  • Relaciones: Asociación, agregación, composición, herencia
  • Multiplicidades: 0..1, 1, 1..*, 0..*, n..m
  • Visibilidad: + (público), - (privado), # (protegido), ~ (paquete)
  • Uso: Análisis y diseño detallado

2. Diagrama de Objetos

Propósito: Mostrar instancias específicas de clases en un momento dado

  • Elementos: Objetos (instancias), enlaces (relaciones)
  • Notación: nombre:Clase con valores concretos
  • Enlaces: Instancias de asociaciones entre objetos
  • Uso: Ejemplificar casos específicos, validar modelo de clases
  • Momento: Instantánea del sistema

3. Diagrama de Componentes

Propósito: Mostrar organización y dependencias entre componentes

  • Elementos: Componentes, interfaces, puertos, conectores
  • Interfaces: Proporcionadas (lollipop) y requeridas (socket)
  • Dependencias: Relaciones entre componentes
  • Manifestación: Artefactos que implementan componentes
  • Uso: Arquitectura de software, modularización

4. Diagrama de Estructura Compuesta

Propósito: Mostrar estructura interna de un clasificador

  • Elementos: Partes, puertos, conectores, colaboraciones
  • Partes: Propiedades con tipo que representan instancias
  • Puertos: Puntos de interacción con el entorno
  • Conectores: Enlaces entre partes y puertos
  • Uso: Arquitectura interna, patrones de diseño

5. Diagrama de Despliegue

Propósito: Mostrar configuración física del sistema

  • Nodos: Recursos computacionales (hardware/software)
  • Artefactos: Elementos físicos del software
  • Manifestaciones: Utilización de artefactos por nodos
  • Protocolos: Comunicación entre nodos
  • Uso: Arquitectura de despliegue, distribución

6. Diagrama de Paquetes

Propósito: Mostrar organización del modelo en paquetes

  • Paquetes: Contenedores de elementos del modelo
  • Dependencias: Relaciones entre paquetes
  • Importación: Import, access (público/privado)
  • Merge: Combinación de contenidos de paquetes
  • Uso: Arquitectura de gran escala, organización

7. Diagrama de Perfiles

Propósito: Definir extensiones específicas de dominio

  • Estereotipos: Extensiones de metaclases UML
  • Valores etiquetados: Propiedades adicionales
  • Restricciones: Reglas en OCL o lenguaje natural
  • Metaclases: Elementos UML que se extienden
  • Uso: Adaptación a dominios específicos

🎭 Diagramas de Comportamiento (7 tipos)

Muestran el comportamiento dinámico del sistema: qué debe suceder en el sistema modelado.

8. Diagrama de Casos de Uso

Propósito: Mostrar funcionalidades del sistema desde perspectiva del usuario

  • Actores: Usuarios, sistemas externos, roles
  • Casos de uso: Funcionalidades del sistema
  • Relaciones: Include, extend, generalización
  • Sistema: Límite del sistema (boundary)
  • Uso: Captura de requisitos, análisis funcional

9. Diagrama de Actividades

Propósito: Mostrar flujo de actividades y decisiones

  • Actividades: Unidades de trabajo
  • Acciones: Pasos atómicos dentro de actividades
  • Flujos: Control y objeto (tokens)
  • Decisiones: Bifurcaciones, uniones, forks
  • Uso: Procesos de negocio, algoritmos

10. Diagrama de Estados

Propósito: Mostrar ciclo de vida de objetos mediante estados

  • Estados: Situaciones en la vida de un objeto
  • Transiciones: Cambios entre estados
  • Eventos: Disparadores de transiciones
  • Acciones: Entry, exit, do activities
  • Uso: Objetos con comportamiento reactivo

11. Diagrama de Secuencia (Interacción)

Propósito: Mostrar interacciones ordenadas temporalmente

  • Líneas de vida: Participantes en la interacción
  • Mensajes: Comunicación entre participantes
  • Activaciones: Períodos de actividad
  • Fragmentos: Alt, opt, loop, par, seq
  • Uso: Escenarios, casos de uso detallados

12. Diagrama de Comunicación (Interacción)

Propósito: Mostrar interacciones enfatizando estructura

  • Objetos: Participantes con roles
  • Enlaces: Conexiones entre objetos
  • Mensajes: Numerados secuencialmente
  • Estructura: Organización espacial
  • Uso: Alternativa a diagramas de secuencia

13. Diagrama de Tiempo (Interacción)

Propósito: Mostrar cambios de estado a lo largo del tiempo

  • Líneas de tiempo: Eje temporal explícito
  • Estados: Valores a lo largo del tiempo
  • Eventos: Puntos de cambio
  • Restricciones: Temporales entre eventos
  • Uso: Sistemas de tiempo real, embedded

14. Diagrama de Vista Global (Interacción)

Propósito: Mostrar flujo de control entre interacciones

  • Referencias: Otros diagramas de interacción
  • Actividades: Nodos del flujo de control
  • Decisiones: Bifurcaciones en el flujo
  • Sincronización: Fork y join
  • Uso: Procesos complejos con múltiples escenarios

🔗 Relaciones en UML

Conexiones semánticas entre elementos del modelo que definen cómo interactúan.

Relaciones Estructurales

  • Asociación (─────): Relación semántica entre clases
  • Agregación (◇────): "Parte de", relación débil
  • Composición (◆────): "Compuesto por", relación fuerte
  • Generalización (────▷): Herencia, "es un tipo de"
  • Realización (┈┈┈▷): Implementación de interfaz

Relaciones de Dependencia

  • Dependencia (┈┈┈>): Uso, relación semántica débil
  • Abstracción: Relación entre niveles de abstracción
  • Uso: Un elemento requiere otro para funcionar
  • Permiso: Acceso a elementos de otro paquete
  • Instanciación: Creación de instancias

🏷️ Estereotipos UML Comunes

Extensiones del vocabulario UML para adaptarse a dominios específicos.

Estereotipos de Clases

  • «interface»: Contrato de operaciones
  • «abstract»: Clase abstracta
  • «enumeration»: Tipo enumerado
  • «utility»: Clase con métodos estáticos
  • «entity»: Clase de dominio persistente
  • «control»: Clase de control
  • «boundary»: Clase de interfaz

Estereotipos de Componentes

  • «executable»: Programa ejecutable
  • «library»: Biblioteca de código
  • «database»: Base de datos
  • «file»: Archivo del sistema
  • «framework»: Marco de trabajo
  • «service»: Servicio web

Métricas de Diseño UML

CBO (Coupling Between Objects) = Σ clases acopladas
DIT (Depth of Inheritance Tree) = máxima profundidad herencia
NOC (Number of Children) = número subclases directas
WMC (Weighted Methods per Class) = Σ complejidad métodos
LCOM (Lack of Cohesion in Methods) = métodos no relacionados

UML 1.0 (1997)

Primera versión estándar por la OMG, unificando metodologías de Booch, Rumbaugh y Jacobson. 9 tipos de diagramas iniciales.

UML 1.4 (2001)

Refinamientos importantes en semántica y notación. Mejora en profiles y constraints.

UML 2.0 (2004)

Revisión mayor: 13 tipos de diagramas, mejor estructura interna, soporte completo para MDA.

UML 2.5 (2015)

Versión actual con 14 tipos de diagramas, simplificaciones y mejoras en especificación.

Proceso Unificado de Software

🔄 Características del Proceso Unificado

Iterativo e incremental, dirigido por casos de uso, centrado en la arquitectura y controlado por riesgos.

Fases del Proceso Unificado

  • Inicio: Visión y viabilidad del proyecto
  • Elaboración: Arquitectura y casos de uso críticos
  • Construcción: Desarrollo incremental
  • Transición: Despliegue y mantenimiento

Disciplinas Principales

  • Modelado de negocio
  • Requisitos
  • Análisis y diseño
  • Implementación
  • Pruebas
  • Despliegue

Disciplinas de Soporte

  • Gestión de configuración y cambios
  • Gestión del proyecto
  • Entorno de desarrollo
  • Métricas y medición

Artefactos Principales

  • Documento de visión
  • Modelo de casos de uso
  • Documento de arquitectura software
  • Modelo de diseño
  • Plan de desarrollo software

Modelado de Requisitos

Identificación de actores, casos de uso y especificación detallada de requisitos funcionales y no funcionales.

Análisis del Sistema

Creación del modelo de análisis con clases conceptuales, colaboraciones y paquetes de análisis.

Diseño Arquitectónico

Definición de la arquitectura software, patrones arquitectónicos y decisiones de diseño principales.

Diseño Detallado

Refinamiento del modelo de diseño con clases de diseño, interfaces y especificaciones detalladas.

Métricas del Proceso Unificado

Progreso = (Casos de Uso Implementados / Total Casos de Uso) × 100
Cobertura = (Requisitos Cubiertos / Total Requisitos) × 100

Patrones de Diseño

🏗️ Gang of Four (GoF)

23 patrones fundamentales clasificados en creacionales, estructurales y de comportamiento, que proporcionan soluciones reutilizables a problemas comunes en el diseño de software orientado a objetos.

Patrones Creacionales

  • Singleton: Instancia única
  • Factory Method: Creación de objetos
  • Abstract Factory: Familias de objetos
  • Builder: Construcción compleja
  • Prototype: Clonación de objetos

Patrones Estructurales

  • Adapter: Interfaz compatible
  • Bridge: Separar abstracción e implementación
  • Composite: Estructura de árbol
  • Decorator: Añadir funcionalidad
  • Facade: Interfaz simplificada
  • Flyweight: Compartir estado intrínseco
  • Proxy: Representante de objeto

Patrones de Comportamiento

  • Observer: Notificación de cambios
  • Strategy: Intercambiar algoritmos
  • Command: Encapsular peticiones
  • State: Comportamiento según estado
  • Template Method: Esquema de algoritmo
  • Visitor: Operaciones en estructura
  • Chain of Responsibility: Cadena de manejadores
  • Mediator: Comunicación entre objetos
  • Memento: Capturar estado interno
  • Iterator: Acceso secuencial
  • Interpreter: Gramática de lenguaje

Principios SOLID

  • SRP: Responsabilidad única
  • OCP: Abierto-cerrado
  • LSP: Sustitución de Liskov
  • ISP: Segregación de interfaces
  • DIP: Inversión de dependencias

Métricas de Calidad de Diseño

Cohesión = (Métodos Relacionados / Total Métodos) × 100
Acoplamiento = Dependencias Externas / Total Clases
Complejidad Ciclomática = Aristas - Nodos + 2

Patrón Singleton

Garantiza una única instancia de una clase y proporciona acceso global a ella. Útil para recursos compartidos como configuraciones o logs.

Patrón Observer

Define dependencia uno-a-muchos entre objetos. Cuando un objeto cambia de estado, notifica automáticamente a todos sus dependientes.

Patrón MVC

Arquitectura que separa la lógica de negocio (Modelo), la presentación (Vista) y el control de flujo (Controlador).

Ejercicios de Diseño Orientado a Objetos

Progreso: 0/10

Ejercicio 1: Métrica CBO (Coupling Between Objects)

Una clase A utiliza directamente las clases B, C, D y E. La clase B utiliza las clases F y G. ¿Cuál es el CBO de la clase A?

CBO(A) = Número de clases con las que A tiene acoplamiento directo

Ejercicio 2: DIT (Depth of Inheritance Tree)

En una jerarquía de herencia: Object → Animal → Mamífero → Felino → Gato. ¿Cuál es el DIT de la clase Gato?

DIT = Profundidad máxima desde la raíz hasta la clase

Ejercicio 3: LCOM (Lack of Cohesion of Methods)

Una clase tiene 5 métodos y 4 atributos. Los métodos M1 y M2 usan atributos A1,A2. M3 usa A3. M4 y M5 usan A4. ¿Cuál es el LCOM4?

LCOM4 = Número de componentes conexas de métodos

Ejercicio 4: Complejidad Ciclomática

Un método tiene 2 bucles for anidados, 3 sentencias if y 1 switch con 4 casos. ¿Cuál es su complejidad ciclomática?

CC = 1 + número de decisiones (if, for, while, case, catch, etc.)

Ejercicio 5: Fan-in y Fan-out

El módulo A es llamado por 3 módulos y llama a 5 módulos. ¿Cuáles son sus valores Fan-in y Fan-out?

Fan-in = Número de módulos que llaman a este módulo
Fan-out = Número de módulos llamados por este módulo

Ejercicio 6: WMC (Weighted Methods per Class)

Una clase tiene 4 métodos con complejidades ciclomáticas 2, 3, 5 y 4 respectivamente. ¿Cuál es el WMC de la clase?

WMC = Σ complejidades ciclomáticas de todos los métodos

Ejercicio 7: NOC (Number of Children)

La clase Animal tiene 3 subclases directas: Mamífero, Ave, Pez. Mamífero tiene 2 subclases: Perro y Gato. ¿Cuál es el NOC de Animal?

NOC = Número de subclases inmediatas (hijos directos)

Ejercicio 8: RFC (Response for Class)

Una clase tiene 6 métodos propios y llama a 8 métodos de otras clases. ¿Cuál es el valor RFC?

RFC = Métodos propios + Métodos llamados desde otras clases

Ejercicio 9: Cálculo de Coste Singleton

Crear una instancia normal cuesta 100ms. Con Singleton se ahorra 80ms por instancia adicional. Si se necesitan 50 instancias, ¿cuál es el ahorro total?

Ahorro = (Instancias - 1) × Coste unitario ahorrado

Ejercicio 10: Eficiencia de Patrones Observer

Un Subject tiene 20 Observers. Si notificar a cada Observer cuesta 5ms, ¿cuánto tiempo total requiere una notificación completa?

Tiempo total = Número de Observers × Tiempo por notificación
