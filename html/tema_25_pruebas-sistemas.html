Introducción a las Pruebas de Sistemas

Las pruebas de sistemas son un proceso fundamental en el desarrollo de software que permite verificar que el sistema funciona correctamente según los requisitos especificados. Este proceso incluye la planificación, ejecución y documentación de diferentes tipos de pruebas.

Objetivos de las Pruebas

  • Detectar defectos antes de la entrega del software
  • Verificar que el sistema cumple los requisitos especificados
  • Validar que el sistema satisface las necesidades del usuario
  • Evaluar la calidad del software desarrollado
  • Proporcionar información sobre el riesgo del software
  • Aumentar la confianza en el sistema

Principios Fundamentales del Testing

1. Las pruebas muestran la presencia de defectos: No pueden probar que no existen errores.

2. Pruebas exhaustivas son imposibles: Se debe usar análisis de riesgo para priorizar.

3. Pruebas tempranas: Detectar errores cuanto antes es más económico.

4. Agrupación de defectos: Los errores tienden a concentrarse en módulos específicos.

5. Paradoja del pesticida: Los mismos casos de prueba repetidos pierden efectividad.

6. Las pruebas dependen del contexto: Se adaptan al tipo de aplicación.

7. Falacia de ausencia de errores: Un sistema sin errores puede no satisfacer necesidades.

Tipos de Pruebas

Existen múltiples clasificaciones de pruebas según diferentes criterios. Es fundamental conocer cada tipo para aplicar la estrategia adecuada.

Clasificación por Niveles

Pruebas Unitarias

Objetivo: Verificar componentes individuales del software.

Responsable: Desarrolladores

Herramientas: JUnit, NUnit, pytest

  • Aislamiento de dependencias
  • Cobertura de código
  • Ejecución automatizada

Pruebas de Integración

Objetivo: Verificar interfaces entre componentes.

Enfoques: Big Bang, Top-down, Bottom-up, Sandwich

Aspectos clave:

  • Comunicación entre módulos
  • Transferencia de datos
  • Interfaces de usuario

Pruebas de Sistema

Objetivo: Verificar el sistema completo integrado.

Ambiente: Similar al productivo

  • Requisitos funcionales
  • Requisitos no funcionales
  • Casos de uso completos

Pruebas de Aceptación

Objetivo: Validar que el sistema satisface necesidades del usuario.

Tipos: Aceptación del usuario, operacional, contractual

  • Criterios de aceptación
  • Escenarios reales
  • Aprobación final

Clasificación por Técnicas

Caja Negra (Black Box)

Pruebas basadas en especificaciones sin conocer la estructura interna.

Técnicas:

  • Partición equivalente
  • Análisis de valores límite
  • Tablas de decisión
  • Transición de estados

Caja Blanca (White Box)

Pruebas basadas en la estructura interna del código.

Criterios de Cobertura:

  • Cobertura de sentencias
  • Cobertura de ramas
  • Cobertura de condiciones
  • Cobertura de rutas

Caja Gris (Grey Box)

Combinación de técnicas de caja negra y caja blanca.

  • Conocimiento parcial del código
  • Pruebas de integración
  • Pruebas de regresión
  • Pruebas de penetración

Pruebas No Funcionales

Verifican aspectos de calidad y rendimiento del sistema.

  • Rendimiento y carga
  • Seguridad
  • Usabilidad
  • Compatibilidad

Documentación de Pruebas

La documentación es fundamental para la trazabilidad, mantenimiento y mejora continua del proceso de pruebas.

Tipos de Documentos

Plan de Pruebas

Documento maestro que define la estrategia general de testing.

  • Objetivos y alcance
  • Estrategia y enfoque
  • Recursos y cronograma
  • Criterios de entrada/salida

Especificación de Casos de Prueba

Detalle específico de cada caso de prueba a ejecutar.

  • Identificador único
  • Precondiciones
  • Pasos de ejecución
  • Resultado esperado

Reporte de Ejecución

Registro de la ejecución de pruebas y sus resultados.

  • Estado de ejecución
  • Resultados obtenidos
  • Defectos encontrados
  • Observaciones

Reporte de Defectos

Documentación detallada de los errores encontrados.

  • Descripción del defecto
  • Pasos para reproducir
  • Severidad y prioridad
  • Estado y asignación

Datos de Prueba

Los datos de prueba son fundamentales para la efectividad del testing. Su selección, generación y gestión requieren estrategias específicas.

Tipos de Datos de Prueba

Datos Válidos

Datos que cumplen con todas las validaciones del sistema.

  • Valores dentro de rangos permitidos
  • Formatos correctos
  • Tipos de datos apropiados
  • Reglas de negocio satisfechas

Datos Inválidos

Datos que violan las reglas de validación del sistema.

  • Valores fuera de rango
  • Formatos incorrectos
  • Tipos de datos erróneos
  • Violación de reglas de negocio

Datos Límite

Valores en los bordes de los rangos permitidos.

  • Valores mínimos y máximos
  • Primero y último de una lista
  • Justo dentro/fuera del rango
  • Valores nulos o vacíos

Datos de Volumen

Grandes cantidades de datos para pruebas de rendimiento.

  • Datasets masivos
  • Múltiples usuarios concurrentes
  • Transacciones simultáneas
  • Stress del sistema

Técnicas de Generación de Datos

Datos de Producción Anonimizados

Uso de datos reales con información sensible enmascarada o eliminada.

Generación Sintética

Creación de datos artificiales que mantienen las características estadísticas.

Datos Aleatorios

Generación automática usando algoritmos para crear valores diversos.

Datos Manuales

Creación específica de datasets para casos de prueba particulares.

Ejercicios Prácticos de Pruebas

Resuelve estos ejercicios para aplicar los conceptos de planificación y ejecución de pruebas.

Progreso: 0/10 ejercicios completados

Ejercicio 1: Un proyecto tiene 150 requisitos funcionales. Se han creado 450 casos de prueba que cubren 120 requisitos. ¿Cuál es el porcentaje de cobertura de requisitos?
Ejercicio 2: En un sistema se encontraron 25 defectos durante las pruebas y 5 defectos adicionales en producción. ¿Cuál es la efectividad de las pruebas?
Ejercicio 3: Un equipo planea ejecutar 200 casos de prueba en 10 días. Si cada tester puede ejecutar 8 casos por día, ¿cuántos testers se necesitan?
Ejercicio 4: Para pruebas de carga, se requiere simular 1000 usuarios concurrentes. Si cada máquina virtual puede simular 50 usuarios, ¿cuántas VMs se necesitan?
Ejercicio 5: Un sistema tiene 10,000 líneas de código. Si se espera una densidad de defectos de 2 defectos por 100 líneas de código, ¿cuántos defectos se estiman encontrar?
Ejercicio 6: En pruebas de integración bottom-up de 5 módulos, se necesitan drivers y stubs. ¿Cuántos stubs se requieren como máximo?
Ejercicio 7: Para una aplicación web con 8 navegadores soportados y 3 sistemas operativos, ¿cuántas combinaciones de pruebas de compatibilidad se requieren?
Ejercicio 8: Un caso de prueba falla el 20% de las veces debido a problemas de timing. Si se ejecuta 15 veces, ¿cuántas veces se espera que falle?
Ejercicio 9: Para cobertura de sentencias del 85% en un módulo de 200 líneas de código ejecutable, ¿cuántas líneas deben ser ejecutadas?
Ejercicio 10: Un proyecto tiene presupuesto de 100,000€ para testing. Si las pruebas cuestan 50€ por caso ejecutado, ¿cuántos casos de prueba se pueden ejecutar?
