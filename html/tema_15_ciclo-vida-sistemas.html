🔄 Fases del SDLC: Cada fase tiene objetivos específicos, entradas definidas, actividades estructuradas y entregables concretos.
1

🎯 Fase de Planificación

Objetivo: Definir el alcance, objetivos y viabilidad del proyecto.

📋 Entradas

  • Solicitud de proyecto
  • Objetivos de negocio
  • Restricciones presupuestarias
  • Recursos disponibles

⚙️ Actividades Principales

  • Estudio de viabilidad
  • Análisis coste-beneficio
  • Identificación de stakeholders
  • Planificación inicial
  • Análisis de riesgos

📤 Entregables

  • Document de viabilidad
  • Plan de proyecto
  • Registro de stakeholders
  • Matriz de riesgos inicial
  • Carta de proyecto

💰 Coste Típico

5-10% del presupuesto total

⏱️ Duración

2-6 semanas según complejidad

👥 Equipo Típico

3-5 personas multidisciplinar

2

📊 Fase de Análisis

Objetivo: Comprender completamente los requisitos funcionales y no funcionales del sistema.

🎯 Análisis de Requisitos

  • Técnicas de elicitación: entrevistas, workshops, observación
  • Casos de uso y historias de usuario
  • Requisitos funcionales vs no funcionales
  • Análisis de procesos de negocio

📋 Documentación

  • Especificación de Requisitos Software (SRS)
  • Modelos de dominio
  • Diagramas de flujo de procesos
  • Matriz de trazabilidad

✅ Validación

  • Revisiones con stakeholders
  • Prototipado conceptual
  • Análisis de factibilidad técnica
  • Aprobación formal de requisitos
⚠️ Riesgos Comunes:
  • Requirements creep: Aumento descontrolado de requisitos
  • Ambigüedad: Requisitos poco claros o contradictorios
  • Stakeholder conflicts: Visiones divergentes no resueltas
  • Gold plating: Funcionalidades innecesarias
3

🏗️ Fase de Diseño

Objetivo: Traducir requisitos en especificaciones técnicas detalladas.

🎨 Diseño de Alto Nivel

  • Arquitectura del sistema: Capas, componentes, patrones
  • Tecnologías: Stack tecnológico, frameworks
  • Interfaces: APIs, integraciones externas
  • Seguridad: Autenticación, autorización, cifrado
  • Infraestructura: Servidores, red, almacenamiento

🔧 Diseño Detallado

  • Modelo de datos: Esquemas, relaciones, índices
  • Diseño de interfaces: Wireframes, mockups, UX
  • Algoritmos: Lógica de negocio compleja
  • Protocolos: Comunicación entre componentes
  • Error handling: Gestión de excepciones
Artefacto de Diseño Propósito Audiencia Herramientas
Diagrama de arquitectura Vista general del sistema Arquitectos, desarrolladores Visio, Lucidchart, Draw.io
Modelo de datos Estructura de información DBA, desarrolladores ERwin, MySQL Workbench
Wireframes/Mockups Diseño de interfaces UX/UI, stakeholders Figma, Sketch, Adobe XD
Diagrama de clases Estructura orientada a objetos Desarrolladores UML tools, IDE

Objetivo: Implementar el sistema según las especificaciones de diseño.

🏗️

Construcción

  • Codificación
  • Pruebas unitarias
  • Code reviews
  • Integración continua
📚

Documentación

  • Documentación técnica
  • Manuales de usuario
  • Comentarios de código
  • APIs documentation
  • Guías de instalación
🔄

Control de Versiones

  • Git workflow
  • Branching strategy
  • Pull requests
  • Release management
  • Tagging y versionado
✅ Mejores Prácticas de Desarrollo:
  • Coding standards: Convenciones de código consistentes
  • DRY Principle: Don't Repeat Yourself
  • SOLID Principles: Diseño orientado a objetos
  • Test-driven development: Pruebas antes que código
  • Refactoring continuo: Mejora constante del código
  • Pair programming: Programación colaborativa

Objetivo: Verificar que el sistema cumple con todos los requisitos y funciona correctamente.

🎯 Hot Tips para Examen - Fases SDLC:
  • Planificación: 5-10% presupuesto, análisis viabilidad
  • Análisis: Elicitación requisitos, SRS, validación
  • Diseño: Alto nivel + detallado, arquitectura + componentes
  • Desarrollo: Codificación + documentación + CI/CD
  • Testing: Unitarias (20%) + Integración (25%) + Sistema (30%) + Aceptación (25%)

Modelo en Cascada

Enfoque: Secuencial y lineal

✅ Ventajas:
  • Simple de entender y gestionar
  • Documentación exhaustiva
  • Funciona bien para proyectos estables
  • Facilita planificación y estimación
❌ Desventajas:
  • Inflexible ante cambios
  • Testing tardío
  • No hay entregables hasta el final
  • Alto riesgo de fracaso
Mejor para: Proyectos bien definidos y estables

RAD (Rapid Application Development)

Enfoque: Desarrollo rápido con prototipado

🎯 Principios RAD
  • Prototipado rápido
  • Desarrollo incremental
  • Timeboxing estricto
  • Participación activa del usuario
  • Reutilización de componentes
⚙️ Fases RAD
  1. Business modeling: Procesos de negocio
  2. Data modeling: Modelo de datos
  3. Process modeling: Funcionalidades
  4. Application generation: Construcción
  5. Testing & turnover: Pruebas y entrega
Limitaciones: Requiere recursos experientes, no escalable para proyectos muy grandes

🏗️ Pirámide de Testing

🔧 Pruebas Unitarias (70%)
  • Alcance: Funciones/métodos individuales
  • Velocidad: Muy rápida (milisegundos)
  • Coste: Muy bajo
  • Mantenimiento: Fácil
  • Herramientas: JUnit, NUnit, Jest
🔗 Pruebas de Integración (20%)
  • Alcance: Interacción entre componentes
  • Velocidad: Rápida (segundos)
  • Coste: Bajo
  • Tipos: Big Bang, Incremental, Sandwich
  • Herramientas: TestNG, Postman, SoapUI
🎭 Pruebas End-to-End (10%)
  • Alcance: Flujos completos de usuario
  • Velocidad: Lenta (minutos)
  • Coste: Alto
  • Fragilidad: Alta
  • Herramientas: Selenium, Cypress, Playwright

🔍 Técnicas de Testing

Tipo Enfoque Técnicas Herramientas
Caja Negra Funcional Equivalencia, valores límite, casos de uso Manual, Automation
Caja Blanca Estructural Cobertura de código, caminos, condiciones Coverage tools, Static analysis
Caja Gris Híbrido Combinación de ambas Testing frameworks
📊 Criterios de Cobertura:
  • Cobertura de sentencias: % líneas ejecutadas
  • Cobertura de ramas: % decisiones evaluadas
  • Cobertura de condiciones: % condiciones booleanas
  • Cobertura de caminos: % rutas de ejecución

🤖 Test Automation

💡 Automation Strategy: No automatizar todo, sino lo que aporta mayor valor y se mantiene fácilmente.

✅ Candidatos para Automatización

  • Pruebas repetitivas y tediosas
  • Pruebas con datos múltiples
  • Pruebas de regresión críticas
  • Pruebas de performance
  • Pruebas que requieren múltiples datasets
  • Smoke tests y sanity checks
ROI Alto: Casos frecuentes y estables

❌ No Automatizar

  • Pruebas que cambian frecuentemente
  • Pruebas de usabilidad subjetiva
  • Pruebas ad-hoc y exploratorias
  • Pruebas que requieren validación visual compleja
  • Casos de uso únicos
  • Pruebas de user experience
ROI Bajo: Coste > Beneficio

🔧 Mantenimiento Correctivo

Objetivo: Corregir defectos encontrados después de la puesta en producción.

🚨 Detección de Defectos

  • Reports de usuarios
  • Monitorización automática
  • Logs de error
  • Auditorías de seguridad

🔍 Análisis y Diagnóstico

  • Reproducción del error
  • Análisis de root cause
  • Impacto assessment
  • Priorización

🛠️ Resolución

  • Desarrollo de fix
  • Testing de regresión
  • Deployment controlado
  • Verificación
🚨 Clasificación de Severidad:
  • Crítica: Sistema no disponible - Fix inmediato
  • Alta: Funcionalidad principal afectada - 24-48h
  • Media: Funcionalidad secundaria - 1-2 semanas
  • Baja: Mejoras menores - Próxima release

  • Coste mantenimiento: Supera 50% coste nuevo desarrollo
  • Tecnología obsoleta: Soporte fabricante descontinuado

  • Performance: No cumple requisitos actuales
  • Compliance: No puede adaptarse a nuevas regulaciones

  • EOL: Migración datos + gestión cambio + archivo
