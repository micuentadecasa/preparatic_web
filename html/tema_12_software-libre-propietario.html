🏛️ Historia del Software Libre

1950s-1960s: Era de Código Abierto

Software se distribuía con código fuente. IBM y académicos compartían programas libremente. No había concepto de "software propietario".

1969: Unbundling de IBM

IBM separa hardware y software. Nace la industria del software comercial. Aparecen primeras licencias propietarias.

1976: Carta Abierta de Bill Gates

Gates critica la "piratería" de software. Defiende el software como propiedad intelectual. Microsoft BASIC como ejemplo.

1983: Proyecto GNU

Richard Stallman inicia GNU (GNU's Not Unix). Objetivo: sistema operativo completamente libre. Nace el concepto de software libre.

1985: Free Software Foundation (FSF)

Stallman funda FSF. Define las 4 libertades del software libre. Desarrollo de herramientas GNU: GCC, Emacs.

1989: GPL v1

Primera GNU General Public License. Concepto de "copyleft" - mantener libertades en derivados.

1991: Linux Kernel

Linus Torvalds libera Linux 0.01. Completaba el sistema GNU/Linux. Modelo de desarrollo distribuido.

1998: Open Source Initiative (OSI)

Eric Raymond y Bruce Perens fundan OSI. Término "Open Source" para evitar confusión con "free". "La Catedral y el Bazar".

2000s: Mainstream Adoption

Apache, MySQL, PHP dominan web. IBM invierte 1.000M USD en Linux. Empresas adoptan OSS masivamente.

2008: GitHub

Plataforma revoluciona colaboración OSS. Git + web + social. 100M+ repositorios públicos.

2014: Microsoft ♥ Open Source

Satya Nadella cambia estrategia. .NET Core open source. Microsoft compra GitHub (2018) por 7.5B USD.

2020s: OSS Everywhere

Cloud, IA, blockchain basados en OSS. Kubernetes domina orquestación. OSS Supply Chain Security crítico.

Fechas clave para examen: GNU Project (1983), GPL v1 (1989), Linux (1991), OSI (1998). Memoriza la cronología de eventos importantes.

💭 Filosofías y Principios

🕊️ Las 4 Libertades del Software Libre (FSF)

📋 Definición Oficial de Richard Stallman

  1. Libertad 0: Ejecutar el programa para cualquier propósito
  2. Libertad 1: Estudiar cómo funciona y adaptarlo (requiere acceso al código fuente)
  3. Libertad 2: Redistribuir copias para ayudar a otros
  4. Libertad 3: Mejorar el programa y hacer públicas las mejoras (requiere acceso al código fuente)

Nota: Si falta cualquier libertad, el software NO es libre.

✅ Libertad 0: Ejecutar

  • Sin restricciones de uso
  • Cualquier propósito: personal, comercial, educativo
  • Sin limitaciones temporales o geográficas
  • Ejemplo violación: "Solo uso no comercial"

🔍 Libertad 1: Estudiar

  • Acceso al código fuente obligatorio
  • Documentación disponible
  • Herramientas de construcción accesibles
  • Formato preferido para modificaciones

🔄 Libertad 2: Redistribuir

  • Copias exactas del original
  • Gratis o con coste (ambos válidos)
  • Cualquier medio: físico, digital
  • Sin pedir permiso al autor

🛠️ Libertad 3: Mejorar

  • Modificaciones permitidas
  • Redistribuir versiones modificadas
  • Beneficio comunitario
  • Código fuente modificado disponible
MNEMOTECNIA: "EEMR" - Ejecutar, Estudiar, Modificar, Redistribuir. Las 4 libertades en orden de más básica a más avanzada.

📖 Definición Open Source (OSI)

📋 10 Criterios Open Source Initiative

  1. Redistribución libre: Sin restricciones de venta o regalo
  2. Código fuente: Disponible y en forma preferida para modificaciones
  3. Trabajos derivados: Permitidos bajo misma licencia
  4. Integridad del autor: Puede requerir cambio de nombre
  5. No discriminación personal/grupal: Todos pueden usar
  6. No discriminación de campos: Cualquier propósito
  7. Distribución de licencia: Derechos automáticos
  8. Licencia no específica de producto: Funciona independiente
  9. Licencia no restringe otro software: No contamina
  10. Licencia neutral tecnológicamente: No favorece interfaces

🤖 Copyleft vs Copyright

Concepto Copyright Copyleft
Objetivo Proteger derechos del autor Proteger libertades del usuario
Restricciones No copiar, modificar, distribuir Mantener misma licencia en derivados
Obras derivadas Prohibidas sin permiso Permitidas con misma licencia
Evolución Fragmentación, versiones propietarias Ecosistema colaborativo, mejoras compartidas
Ejemplos Windows, Office, Photoshop Linux, GCC, WordPress
CONCEPTO CLAVE: Copyleft usa copyright para garantizar libertad. "Copyright para eliminar copyright". Paradoja legal genial.

🏛️ Modelos Filosóficos

🏰 Modelo Catedral

  • Desarrollo centralizado por pequeño grupo
  • Planificación extensa antes de lanzar
  • Releases espaciados en el tiempo
  • Control estricto de calidad
  • Ejemplos: Windows, Oracle, Emacs (tradicional)

🏪 Modelo Bazar

  • Desarrollo distribuido masivo
  • "Release early, release often"
  • Peer review automático
  • "Many eyeballs make bugs shallow"
  • Ejemplos: Linux, Apache, GNU/Linux distros
# Principios del Modelo Bazar (Eric Raymond) 1. "Todo buen software empieza por scratch personal del desarrollador" 2. "Los buenos programadores saben qué escribir. Los grandes saben qué reescribir (y reutilizar)" 3. "Si no te diviertes programando, estás haciendo algo mal" 4. "Dada suficiente base de beta-testers y co-desarrolladores, casi todo problema será caracterizado rápidamente y la solución será obvia para alguien" # Ley de Linus: "With enough eyeballs, all bugs are shallow"

🌍 Impacto Social y Ético

🎯 Argumentos Éticos del Software Libre

  • Soberanía tecnológica: No dependencia de proveedores
  • Brecha digital: Acceso universal a tecnología
  • Transparencia: Auditabilidad de sistemas críticos
  • Educación: Aprendizaje sin restricciones
  • Innovación colectiva: Bien común vs lucro individual
  • Privacidad: Control sobre datos personales

⚠️ Críticas al Software Propietario

  • Vendor lock-in: Dependencia tecnológica
  • Obsolescencia programada: Forzar actualizaciones
  • Falta de transparencia: Cajas negras
  • Costes de licencias: Barrera económica
  • Formatos propietarios: Interoperabilidad limitada
  • Surveillance capitalism: Monetización de datos
STALLMAN vs TORVALDS: Stallman enfoque ético-filosófico (software debe ser libre). Torvalds enfoque pragmático (el mejor software gana). Ambas visiones coexisten.

📜 Licencias de Software

🎯 Taxonomía de Licencias

Tipo Características Copyleft Ejemplos Proyectos
Strong Copyleft Derivados deben ser libres Fuerte GPL v2/v3, AGPL Linux, GCC, WordPress
Weak Copyleft Solo librerías dinámicas Débil LGPL, MPL, EPL Qt, Firefox, Eclipse
Permissive Máxima libertad, sin copyleft No MIT, BSD, Apache React, Node.js, Django
Public Domain Sin derechos reservados No aplicable Unlicense, CC0 SQLite, algunos algoritmos
Proprietary Todos los derechos reservados No aplicable EULA comerciales Windows, Office, Oracle DB

⚖️ Licencias Copyleft Principales

GPL v2

  • Año: 1991
  • Copyleft fuerte: Derivados GPL
  • Linking: Estático y dinámico
  • Distribución: Código fuente obligatorio
  • Proyectos: Linux kernel, MySQL, WordPress
  • Incompatible con: GPL v3

GPL v3

  • Año: 2007
  • Anti-tivoización: Claves de instalación
  • Patentes: Protección explícita
  • DRM: Restricciones anti-circunvención
  • Proyectos: GCC, Bash, GIMP
  • Incompatible con: GPL v2

AGPL v3

  • Año: 2007
  • Network copyleft: SaaS incluido
  • Cloud loophole: Cerrado
  • Distribución web: Código disponible
  • Proyectos: MongoDB, NextCloud
  • Controversia: Empresas evitan

LGPL v3

  • Lesser GPL: Copyleft débil
  • Linking dinámico: No contagia
  • Librerías: Uso en software propietario
  • Modificaciones LGPL: Deben ser libres
  • Proyectos: Qt, GTK+, VLC
GPL v2 vs GPL v3: Principal diferencia es tivoización. GPL v3 requiere claves/métodos para instalar versiones modificadas en hardware cerrado.

🆓 Licencias Permisivas

MIT License

  • Más simple y popular
  • Sin copyleft: Derivados pueden ser propietarios
  • Solo requiere: Copyright notice
  • Compatible con: Todo
  • Proyectos: React, Node.js, jQuery
Copyright (c) 2024 Autor Permission is hereby granted, free of charge, to any person obtaining a copy of this software and associated documentation files (the "Software"), to deal in the Software without restriction, including without limitation the rights to use, copy, modify, merge, publish, distribute, sublicense, and/or sell copies of the Software...

Apache 2.0

  • Protección patentes explícita
  • Contributor License Agreement
  • Attribution requerida
  • NOTICE file obligatorio
  • Proyectos: Apache HTTP, Android, Kubernetes

BSD 3-Clause

  • Original Berkeley Software Distribution
  • 3 cláusulas: Copyright, disclaimer, endorsement
  • No endorsement sin permiso
  • Proyectos: FreeBSD, nginx, Django

ISC License

  • Internet Systems Consortium
  • MIT simplificada
  • Menos texto legal
  • Proyectos: OpenBSD, npm packages

🔄 Compatibilidad de Licencias

📋 Matriz de Compatibilidad (Simplificada)

Licencia A\B MIT Apache 2.0 GPL v2 GPL v3 LGPL
MIT ✅ ✅ ✅ ✅ ✅
Apache 2.0 ✅ ✅ ❌ ✅ ✅
GPL v2 ✅ ❌ ✅ ❌ ✅

REGLA PRÁCTICA: Permisivas (MIT, BSD) compatibles con todo. GPL más restrictiva. GPL v2 y v3 incompatibles entre sí. Apache 2.0 incompatible con GPL v2.

⚠️ Licencias Problemáticas

❌ Licencias NO Libres (pero populares)

  • JSON License: "Shall be used for Good, not Evil" (discrimina uso)
  • CC BY-NC: No commercial use (discrimina campo de uso)
  • SSPL: MongoDB license (controvertida, no OSI)
  • Business Source License: Eventually open (no inmediatamente libre)
  • Shared Source: Microsoft (solo lectura, no modificación)

💼 Modelos de Negocio

💰 Estrategias de Monetización OSS

🎯 Open Core

  • Núcleo open source + características premium
  • Dual licensing: OSS + commercial
  • Ejemplos: MySQL, GitLab, Elastic
  • Ventajas: Adoption masiva + revenue
  • Desventajas: Community friction

🛠️ Professional Services

  • Support + training + consulting
  • Implementation + customization
  • Ejemplos: Red Hat, Canonical, SUSE
  • Ventajas: Recurring revenue
  • Escalabilidad: Limitada por headcount

☁️ SaaS/Hosted

  • Software as a Service del OSS
  • Management + infrastructure
  • Ejemplos: GitHub, GitLab.com, MongoDB Atlas
  • Ventajas: Recurring, scalable
  • Competencia: AWS managed services

📦 Commercial Licensing

  • Dual license: GPL + proprietary
  • Enterprise features solo comercial
  • Ejemplos: Qt, MySQL (histórico)
  • Ventajas: Revenue directo
  • Complejidad: Gestión legal

🔧 Hardware Integration

  • OSS + hardware optimizado
  • Appliances especializados
  • Ejemplos: Raspberry Pi, System76
  • Ventajas: Diferenciación
  • Mercado: Más pequeño

🎓 Training & Certification

  • Courses + certification programs
  • Expert knowledge monetization
  • Ejemplos: LPI, Red Hat Training
  • Ventajas: High margins
  • Escalabilidad: Online courses
