📖 Resumen General

🎯 ¿Qué es la Gestión de Configuración?

La Gestión de Configuración (CM) es un proceso sistemático para controlar los cambios en el software y hardware, asegurando la integridad, trazabilidad y consistencia del sistema a lo largo de su ciclo de vida.

🔍 Objetivos Principales

  • Control de cambios: Gestionar modificaciones de forma controlada
  • Integridad del sistema: Mantener consistencia entre componentes
  • Trazabilidad: Registro completo de todos los cambios
  • Reproducibilidad: Capacidad de recrear cualquier versión
  • Automatización: Reducir errores manuales y aumentar eficiencia
  • Compliance: Cumplir requisitos normativos y de auditoría

📊 Beneficios Clave

  • Reducción de riesgos: Control sobre cambios críticos
  • Mayor calidad: Procesos estandarizados y validados
  • Eficiencia operacional: Automatización de tareas repetitivas
  • Cumplimiento normativo: Auditorías y controles documentados
  • Recuperación rápida: Rollback automático ante fallos
  • Colaboración mejorada: Visibilidad de cambios entre equipos

🔧 Fundamentos de la Gestión de Configuración

📦 Elementos de Configuración (CIs)

Los Configuration Items son los componentes individuales de un sistema que están bajo control de gestión de configuración.

  • Hardware: Servidores, routers, switches, dispositivos de almacenamiento
  • Software: Aplicaciones, sistemas operativos, middleware, bases de datos
  • Documentación: Manuales, procedimientos, especificaciones técnicas
  • Personal: Roles, responsabilidades, competencias
  • Servicios: Servicios IT, procesos de negocio
  • Ubicaciones: Centros de datos, oficinas, sitios remotos

🔗 Relaciones entre CIs

  • Dependencias: Un CI requiere otro para funcionar
  • Conectividad: CIs conectados física o lógicamente
  • Ubicación: CIs en la misma localización física
  • Uso: Un CI utiliza los servicios de otro
  • Propiedad: Relación de pertenencia o responsabilidad

Identificación

Catalogación de todos los elementos de configuración del sistema, estableciendo una línea base completa.

Control

Establecimiento de procedimientos para gestionar cambios en los elementos de configuración.

Contabilidad del Estado

Registro y reporte del estado actual de todos los elementos y sus cambios históricos.

Verificación y Auditoría

Validación de que la configuración real coincide con la documentada y cumple requisitos.

📋 Control de Cambios

🔄 Proceso de Gestión de Cambios

Proceso estructurado para evaluar, aprobar, implementar y revisar cambios en el sistema.

📝 Flujo de Trabajo de Cambios

1

Solicitud de Cambio (RFC)

Iniciación formal del cambio con justificación y detalles técnicos

2

Evaluación de Impacto

Análisis de riesgos, recursos necesarios y sistemas afectados

3

Aprobación

Revisión por CAB (Change Advisory Board) y autorización

4

Planificación

Programación del cambio, asignación de recursos y preparación

5

Implementación

Ejecución del cambio siguiendo el plan establecido

6

Validación

Verificación de que el cambio cumple objetivos y no causa problemas

7

Cierre

Documentación de resultados y actualización de registros

🎯 Change Advisory Board (CAB)

  • Change Manager: Responsable del proceso de cambios
  • Representantes técnicos: Expertos en sistemas afectados
  • Propietarios de servicios: Responsables de servicios de negocio
  • Seguridad: Evaluación de impacto en seguridad
  • Riesgos: Análisis y mitigación de riesgos
  • Usuario final: Representación de usuarios impactados

🏗️ Estrategias de Branching

  • Git Flow: Modelo con ramas master, develop, feature, release, hotfix
  • GitHub Flow: Flujo simplificado con rama main y feature branches
  • GitLab Flow: Combina feature-driven y environment branches
  • Trunk-based Development: Desarrollo directo en rama principal
# Ejemplo de workflow Git # Crear nueva funcionalidad git checkout -b feature/nueva-funcionalidad git add . git commit -m "feat: implementar nueva funcionalidad" # Fusionar a develop git checkout develop git merge feature/nueva-funcionalidad # Crear release git checkout -b release/v2.1.0 git tag -a v2.1.0 -m "Release version 2.1.0" # Deploy a producción git checkout master git merge release/v2.1.0 git push origin master --tags

🧮 Ejercicios Prácticos

1. Si un equipo implementa 50 cambios por mes y 8 fallan, ¿cuál es la tasa de fallos (Change Failure Rate)?
2. Calcular el tiempo de despliegue si se procesan 120 cambios en 40 horas de ventana de mantenimiento.
3. Si el MTTR es 2 horas y se producen 24 incidentes al año, ¿cuántas horas totales de downtime hay?
4. Calcular el ROI de automatización: inversión 100.000€, ahorro anual 35.000€, período 5 años.
5. Si un repositorio tiene 1500 commits y 12 releases, ¿cuántos commits promedio por release?
6. Calcular la eficiencia de automatización si de 200 tareas mensuales, 170 son automatizadas.
7. Si el Lead Time promedio es 6 días y se quiere reducir a 2 días, ¿cuál es la mejora porcentual?
8. Calcular el número de versiones si se liberan cada 2 semanas durante 1 año (52 semanas).
9. Si el costo manual de un despliegue es 500€ y el automatizado 50€, ¿cuál es el ahorro por despliegue?
10. Calcular la disponibilidad si el sistema funciona 8640 horas de 8760 horas anuales.
