🌐 Modelo de Dominio

Definición: Modelo conceptual que describe un problema específico representando el vocabulario y los conceptos clave a través de entidades, atributos, papeles y relaciones.

  • Utiliza clases conceptuales sin métodos
  • Se enfoca en comprender el problema
  • No contempla detalles de implementación
  • Representa el vocabulario del dominio

🎯 Orientación a Objetos

Definición AOO: "Método de análisis que examina los requisitos desde la perspectiva de las clases y objetos encontrados en el vocabulario del dominio del problema"

✅ Ventajas de la OO

  • Modelos similares al mundo real
  • Facilita el desarrollo de sistemas complejos
  • Aumenta la reutilización de software
  • Desarrollo iterativo de aplicaciones
  • Interoperabilidad entre aplicaciones
  • Sistemas más pequeños y eficientes
  • Escalable para aplicaciones de cualquier tamaño

🏗️ Características del Diseño OO

  • Modularidad: Separación en componentes
  • Extensibilidad: Fácil modificación
  • Ocultación: Information hiding
  • Integrable: Componentes reutilizables
  • Acoplamiento débil: Mínima dependencia
  • Cohesión fuerte: Elementos relacionados

📚 Metodologías de Diseño OO

Metodología Autor Enfoque Descripción

🏛️ Principios SOLID

Single Responsibility

Responsabilidad única: Una clase debe tener una sola razón para cambiar, es decir, una única responsabilidad contenida en la clase.

Open/Closed

Abierto/Cerrado: Las clases deben estar abiertas para extensión, pero cerradas para modificación.

Liskov Substitution

Sustitución de Liskov: Los objetos deben ser reemplazables por instancias de sus subtipos sin alterar el funcionamiento.

Interface Segregation

Segregación de interfaces: Muchas interfaces cliente específicas son mejores que una interfaz general.

Dependency Inversion

Inversión de dependencias: Depender de abstracciones, no de implementaciones. IoC e Inyección de Dependencias.

📋 Diagramas de Clases

Definición: Tipo de diagrama de estructura estática en UML que describe la estructura de un sistema mostrando las clases, sus atributos, operaciones y las relaciones entre los objetos.

🏗️ Elementos Principales

📦 Clases

Elemento principal del diagrama para representar conceptos o entidades.

+--------------------------------+ | NombreClase | +--------------------------------+ | - atributo1: Tipo | | - atributo2: Tipo | +--------------------------------+ | + método1(): TipoRetorno | | + método2(param): void | +--------------------------------+

🔗 Relaciones

Identifican dependencias entre clases con nombre y multiplicidad.

  • Asociación: Línea simple
  • Agregación: Diamante vacío
  • Composición: Diamante relleno
  • Herencia: Flecha vacía
  • Dependencia: Flecha discontinua

🔐 Tipos de Acceso

Símbolo Tipo Acceso
+ Público Desde cualquier clase
- Privado Solo desde la propia clase
# Protegido Clase y clases derivadas

🔄 Tipos de Relaciones

🔗 Asociación

Representación: Línea simple

Tipo de relación más común, representa dependencia semántica

◇ Agregación

Representación: Diamante vacío

Relación dinámica "parte-todo". El objeto base puede tener multiplicidad > 1

◆ Composición

Representación: Diamante relleno

Relación estática "parte-todo". Objeto base con multiplicidad 1

△ Herencia

Representación: Flecha vacía

Apunta hacia la clase padre. Relación "es-un"

⟶ Dependencia

Representación: Flecha discontinua

Una clase requiere otra para ofrecer sus funcionalidades

🗂️ Modelo Entidad-Relación

Definición: Representa el Modelo conceptual de datos. Es único para cada sistema de información y se corresponde con el nivel conceptual de la Arquitectura ANSI a tres niveles.

🏢 Entidades

Objeto real o abstracto de interés en una organización.

📋 Características

  • Debe tener existencia propia
  • Cada ocurrencia debe poder distinguirse
  • Mismos tipos de características (atributos)
  • Proceso iterativo de definición
┌─────────────────┐ │ CLIENTE │ └─────────────────┘

Representación de una entidad

🏷️ Atributos

Propiedades o características de un tipo de entidad.

📝 Tipos de Atributos

  • Simple: No divisible
  • Compuesto: Subdividido en partes
  • Univaluado: Un solo valor
  • Multivaluado: Múltiples valores
  • Obligatorio: Debe tener valor
  • Derivado: Calculado de otros

🔑 Tipos de Claves

  • Superclave: Conjunto de atributos que distingue entidades
  • Clave candidata: Superclave mínima
  • Clave primaria: Clave candidata seleccionada
  • Principio de unicidad: Mínimos atributos posibles

🔗 Relaciones o Interrelaciones

Asociación entre entidades caracterizada por restricciones específicas.

🎯 Grado de Relaciones

  • Unitarias/Unarias: Una entidad se relaciona consigo misma
  • Binarias: Entre dos entidades
  • Ternarias: Entre tres entidades
  • N-arias: Entre N entidades
  • Reflexivas: Caso especial de unarias

📊 Tipos de Correspondencia

1:1

Una ocurrencia se asocia con máximo una de la otra entidad

1:N

Una ocurrencia se asocia con muchas de la otra, pero no viceversa

N:M

Múltiples ocurrencias se asocian en ambas direcciones

Ejemplo de Relación N:M

┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ PROFESOR │────N────│ IMPARTE │────M────│ ASIGNATURA │ └─────────────┘ └─────────────┘ └─────────────┘

🔒 Entidades Débiles

Entidades que dependen de otra para su existencia o identificación.

🆔 Dependencia por Identificador

La entidad débil no tiene suficientes atributos para formar su clave primaria.

  • Necesita la PK de la entidad fuerte
  • No garantiza principio de unicidad
  • Siempre corresponde a relación 1:N

🔄 Dependencia por Existencia

La entidad débil tiene PK suficiente pero depende de otra para existir.

  • Mejor solución que d.p.i.
  • Asegura PK siempre válido
  • Entidad débil puede ser eliminada sin afectar fuerte

🚀 Modelo E/R Extendido

📏 Cardinalidad

Número mínimo y máximo de ocurrencias que pueden participar en una relación.

  • Obligatoria: Mínimo 1
  • Opcional: Mínimo 0

🏗️ Jerarquías

Generalización: De lo concreto a lo abstracto

Especialización: De lo general a lo particular

  • Cobertura Total: Todos los elementos están en subentidades
  • Cobertura Parcial: Algunos elementos no están en subentidades
  • Disjunta: Un elemento en máximo una subentidad
  • Con Solapamiento: Un elemento puede estar en varias subentidades

📊 Transformación a Tablas

  • 1:N: Se propaga clave de cardinalidad 1 a N
  • 1:1: Caso particular de 1:N, propagación bidireccional
  • N:M: Requiere tabla intermedia con claves de ambas entidades
  • Entidades débiles: PK formada por concatenación de claves

❓ Preguntas Frecuentes

¿Cuál es la diferencia entre modelo conceptual y lógico?

Modelo Conceptual (E/R): Representa la realidad del problema independientemente de la tecnología. Es único para cada sistema.

Modelo Lógico (DED): Depende del SGBD a utilizar. Se deriva del conceptual pero considera limitaciones tecnológicas.

¿Cuándo usar enfoque estructurado vs orientado a objetos?

Estructurado: Sistemas donde datos y procesos están claramente separados, aplicaciones de procesamiento batch, sistemas legacy.

Orientado a Objetos: Sistemas complejos, aplicaciones interactivas, cuando se requiere reutilización, desarrollo iterativo.

¿Cómo decidir entre agregación y composición?

Agregación: Cuando las partes pueden existir independientemente del todo (ej: Profesor-Departamento).

Composición: Cuando las partes no tienen sentido sin el todo (ej: Casa-Habitación).

¿Qué es mejor: dependencia por identificador o por existencia?

Dependencia por existencia es generalmente mejor porque:

  • La entidad débil tiene su propia PK completa
  • Mejor rendimiento en consultas
  • Mayor flexibilidad para cambios futuros
  • Cumple principio de unicidad

¿Cómo transformar relaciones N:M a tablas?

Las relaciones N:M siempre requieren una tabla intermedia:

  • La tabla intermedia contiene las PK de ambas entidades
  • Estas PK forman la PK compuesta de la tabla intermedia
  • Los atributos de la relación van en la tabla intermedia
  • Se establecen FK hacia las tablas originales

💡 Mejores Prácticas

🎯 Diseño de E/R

  • Identifica entidades principales primero
  • Define atributos mínimos necesarios
  • Establece relaciones claras y precisas
  • Normaliza para evitar redundancia
  • Documenta restricciones de integridad

🔧 Modelado OO

  • Aplica principios SOLID consistentemente
  • Favorece composición sobre herencia
  • Define interfaces claras y estables
  • Implementa patrones de diseño apropiados
  • Mantén bajo acoplamiento, alta cohesión

📝 Glosario de Términos

Abstracción: Proceso de ocultar detalles innecesarios y mostrar solo las características esenciales de un objeto.
Agregación: Relación "parte-todo" donde las partes pueden existir independientemente del todo.
Cardinalidad: Número mínimo y máximo de instancias que pueden participar en una relación.
Composición: Relación "parte-todo" fuerte donde las partes no pueden existir sin el todo.
Encapsulación: Principio de ocultar la implementación interna y exponer solo la interfaz necesaria.
Entidad débil: Entidad que depende de otra para su existencia o identificación completa.
Herencia: Mecanismo que permite que una clase derive propiedades y comportamientos de otra.
Multiplicidad: Notación que especifica cuántas instancias de una clase pueden estar relacionadas.
Polimorfismo: Capacidad de un mismo mensaje de producir diferentes comportamientos según el objeto receptor.
UML: Lenguaje de Modelado Unificado para especificar, visualizar y documentar sistemas software.
